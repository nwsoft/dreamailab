## 1. 단일 진실 공급원(SSOT) 계층

| 우선순위 | 대상 | 용도 |
|----------|------|------|
| 1 | `app/services/*/page.tsx`, `app/ir`, `app/business`, `app/technology`, `app/page.tsx` | **투자·B2B·기관·외부 방문자**가 보는 최종 메시지 |
| 2 | `docs/services/*.md` | 위 웹과 동기화하는 **개발·검수 스펙** |
| 3 | `app/news/data.ts` | **보도·홍보·검색·내부 아카이브** (시점별 카피 보존) |
| 4 | 루트 `드림에이아이랩_*.md` | 레거시 참고. **대외 기준으로 사용 금지** |

**원칙:** IR·과제·계약·파트너 미팅에서 인용하는 수치·상태·법적 성격의 문구는 **1·2만** 인용한다.

## 2. 뉴스센터 미디어 허브 — 6개 카테고리 (2026-07-07)

뉴스센터는 **모든 글을 보도자료로 쓰지 않는다**. 각 글은 아래 채널 중 **하나**에만 속한다.

| 카테고리 | `value` | 주인공 | 용도 |
|----------|---------|--------|------|
| 회사 뉴스 | `company` | 회사의 변화 | 오픈·배포·MOU·투자·채용·파트너십 |
| 서비스 업데이트 | `updates` | 새 기능·요금제 | 기능 추가, 로드맵, 요금제 변경 |
| 기술 블로그 | `tech` | 기술 | AI디지털케어로그, RWE, OCR, 아키텍처 |
| 인사이트 | `insight` | 문제·시장·철학 | 사회 문제, 왜 필요한가, 사례·분석 |
| Founder Story | `founder` | 창업자 | 정해성 CTO 인터뷰, 왜 만들었는가 |
| 보도자료 | `press` | 서비스·사실 | 언론 배포용 공식 발표 (감성·장사례 최소) |

**SSOT:** 카테고리 정의·기사 ID 매핑 → `lib/news-categories.ts` (`NEWS_ARTICLE_CATEGORY_BY_ID`)  
**재분류 스크립트:** `node scripts/recategorize-news.mjs`

### 반복 금지 (신규·수정 기사)

- **정해성 CTO**·**AI디지털케어로그** 장문은 `founder`·`tech` 또는 기술 소개 페이지에만.
- `updates`·`press`·`company`·`insight` 하단에는 필요 시 **「기술 배경」** 1~2문장만.

### SEO·LLM 신뢰도

- 채널별 **일관된 문체·구조** → 크롤러·요약 모델이 콘텐츠 유형을 구분하기 쉬움 (E-E-A-T·주제 명확성).
- `llms.txt`, JSON-LD `articleSection`, 카테고리 필터 URL을 함께 쓰면 효과 증대.
- 모든 글을 `press`로 두면 검색·인용 시 「보도자료」와 「블로그」가 혼재되어 **신뢰도·스니펫 품질**이 떨어질 수 있음.

## 3. 뉴스 아카이브의 「세계 최초」「국내 최초」 등 — 문제가 되는가?

### 결론: **역할을 나누면 대체로 문제 없음**

| 관점 | 판단 |
|------|------|
| **법·광고 심의** | 뉴스/보도자료 형식의 **과거 시점 홍보 카피**는 통상 “당시 보도 문맥”으로 이해. 다만 **현재 웹 메인·IR·서비스 페이지**에 동일 문구를 그대로 올리면 리스크가 커진다. |
| **투자·VC·TIPS 심사** | 심사위원이 **뉴스만** 보고 판단하면 과장으로 읽힐 수 있음 → **IR/서비스 각주·Status**로 현재 상태를 먼저 보게 하는 것이 안전. |
| **SEO·검색** | 롱테일·브랜드 검색용으로 아카이브 유지 **가치 있음**. 삭제보다 **채널 라벨·메타**로 구분. |
| **내부 자료** | 팀·홍보·언론 대응용 **타임라인**으로 유지 적절. |

### 권장 (뉴스 쪽 최소 정비 — 전면 수정 아님)

1. 뉴스 목록/상세에 **채널 배지** 유지·강화: `보도자료` / `서비스 업데이트` / `내부·홍보` 등 (이미 카테고리 있으면 UI만 명확히).
2. IR·비즈니스·서비스 페이지 하단 FAQ 또는 각주에 한 줄:  
   *「보도·뉴스 아카이브는 당시 홍보·보도 문맥이며, 제품·사업 현황은 각 서비스 페이지 및 IR을 기준으로 합니다.」*
3. **뉴스 본문 일괄 톤 다운은 하지 않는다** (아카이브 훼손·검색 손실). 필요 시 **신규 기사**만 SSOT 톤으로 작성. 뉴스 상세에는 **아카이브 안내 + SSOT 링크** 배너를 표시한다 (2026-07-06 반영).

## 4. 모순으로 보이는 대표 케이스 — 실제 맥락

| 현상 | 맥락 | 처리 |
|------|------|------|
